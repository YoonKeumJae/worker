# Worker 프로젝트 명세

## 서비스 개요

`worker`는 업무 중 반복적으로 필요한 작은 작업들을 로컬에서 처리하는 macOS용 GUI 앱이다.

첫 버전은 URL을 입력받아 QR코드를 생성하고, 파일 저장과 이미지 클립보드 복사를 로컬에서 처리하는 기능에 집중한다. 현재는 이미지 파일 형식 변환 기능도 로컬 처리 도구로 제공한다. 이후 Windows 환경에서도 한글 파일명이 깨지지 않도록 처리하는 작업 등 다양한 도구 기능을 확장할 수 있는 구조를 목표로 한다.

## 목표 사용자

- 개인 사용자
- 업무 중 자주 필요한 파일/문자열/이미지 관련 작업을 빠르게 처리하려는 사용자
- 서버나 계정 없이 로컬 앱에서 작업을 끝내고 싶은 사용자

## 문제 정의

업무 중에는 작지만 반복적인 작업이 자주 발생한다.

- URL을 QR코드로 바꾸기
- 이미지 파일 형식 바꾸기
- 한글 파일명이 Windows 환경에서 깨지지 않도록 처리하기
- 파일이나 텍스트를 목적에 맞게 빠르게 변환하기

각 작업마다 별도 웹사이트나 도구를 찾으면 시간이 낭비되고, 민감한 파일이나 텍스트를 외부 서비스에 업로드해야 하는 위험도 생긴다. `worker`는 이런 작업을 로컬 앱 안에서 처리한다.

## 첫 버전 범위

첫 버전은 QR코드 생성 기능과 이미지 포맷 변환 기능을 제공한다.

### QR코드 생성

- 사용자는 URL을 입력한다.
- 앱은 입력된 URL을 QR코드 이미지로 생성한다.
- 생성된 QR코드는 화면에서 확인할 수 있어야 한다.
- 생성된 QR코드는 PNG 파일로 저장할 수 있어야 한다.
- 생성된 QR코드는 SVG 파일로 저장할 수 있어야 한다.
- 생성된 QR코드 이미지는 클립보드로 복사할 수 있어야 한다.
- PNG 저장, SVG 저장, 이미지 클립보드 복사는 `흰색` 또는 `투명` 배경을 선택할 수 있어야 한다.
- 배경 기본값은 기존 호환을 위해 `흰색`이어야 한다.

### 이미지 포맷 변환

- 사용자는 하나 이상의 이미지 파일을 선택한다.
- 사용자는 대상 포맷을 PNG, JPG, HEIC, WebP 중 선택한다.
- 앱은 선택 파일을 대상 포맷으로 변환한다.
- 앱은 새 복사본을 남기지 않고 원본 파일을 대상 확장자 파일로 교체한다.
- 대상 파일명이 기존 파일과 충돌하면 변환을 시작하지 않아야 한다.
- 이미 대상 포맷인 파일은 재인코딩하지 않고 건너뛰어야 한다.
- JPG처럼 투명도를 저장하지 않는 포맷은 투명 픽셀을 흰색 배경으로 합성한다.
- 애니메이션 WebP는 프레임 손실 방지를 위해 변환하지 않아야 한다.
- 애니메이션 PNG(APNG)는 프레임 손실 방지를 위해 변환하지 않아야 한다.
- HEIC sequence는 프레임 손실 방지를 위해 변환하지 않아야 한다.

## 향후 확장 후보

- Windows 호환 한글 파일명 처리
- 텍스트 변환 도구
- 파일명 일괄 변경
- 이미지 크기 변경
- 문서/이미지 압축
- 기타 반복 업무 자동화 도구

## 비기능 요구사항

- 서버가 필요 없는 로컬 앱이어야 한다.
- 기본 대상은 macOS GUI 앱이다.
- 향후 Windows 버전 출시 가능성을 고려한다.
- 기능 확장이 쉬운 구조여야 한다.
- 개인 데이터와 작업 대상 파일은 외부로 전송하지 않는다.
- 네트워크가 없어도 기본 기능은 동작해야 한다.
- 작업 실패 시 사용자가 이해할 수 있는 오류 메시지를 보여준다.
- 파일을 변경하는 기능은 원본 손상 위험을 줄이는 방식으로 설계한다.

## 기술 스택

기술 스택은 다음으로 확정한다.

- 앱 프레임워크: Tauri 2
- UI: React
- 언어: TypeScript
- 빌드 도구: Vite
- 로컬 작업 엔진: Rust
- 패키지 매니저: pnpm

선택 이유:

- macOS GUI 앱 개발 가능
- 향후 Windows 앱 확장 가능
- 서버 없이 로컬 동작 가능
- 파일 처리와 이미지 처리 기능 구현 용이
- 기능별 모듈화 쉬움
- 설치와 배포가 개인 사용자에게 부담스럽지 않음
- Electron 대비 앱 크기와 리소스 부담이 작음
- Rust 기반 로컬 작업 처리로 안정성과 성능 확보 가능

## 아키텍처 원칙

- UI와 작업 로직을 분리한다.
- 각 기능은 독립 모듈로 추가할 수 있게 설계한다.
- 공통 기능은 재사용 가능한 유틸리티로 분리한다.
- 파일 처리 기능은 입력, 처리, 출력 단계를 명확히 나눈다.
- 기능 추가 시 기존 기능을 깨뜨리지 않도록 테스트 가능한 구조를 유지한다.
- 외부 라이브러리는 필요성과 유지보수성을 검토한 뒤 도입한다.

## 제외 범위

첫 버전에서는 다음을 제외한다.

- 서버 기능
- 사용자 계정
- 로그인
- 클라우드 저장
- 결제
- 팀 협업 기능
- Windows 앱 출시

Windows 앱은 첫 버전 제외 범위지만, 기술 선택과 구조 설계에서 향후 확장 가능성을 고려한다.

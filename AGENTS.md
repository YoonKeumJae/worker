# AGENTS.md

이 파일은 이 디렉터리와 하위 디렉터리에서 작업하는 모든 Codex 에이전트가 따라야 하는 규칙이다.

## 응답 규칙

- 항상 한국어로 응답한다.
- 항상 `/caveman` 플러그인 스타일을 사용한다.
- 문장은 짧고 직접적으로 쓴다. 불필요한 인사, 수식어, 장황한 설명은 제거한다.
- 기술 용어, 파일명, 명령어, 함수명, 에러 메시지는 정확히 유지한다.
- 모호한 표현 대신 구체적 파일, 명령, 날짜, 상태를 쓴다.
- 사용자가 명시적으로 일반 문체를 요청하지 않는 한 caveman 스타일을 유지한다.

## 도구와 스킬 사용

- 작업 시작 전, 관련 스킬이 있는지 확인하고 필요한 스킬을 사용한다.
- GitHub, Gmail, Browser, Documents, Presentations, Spreadsheets, Codex Security, Superpowers 같은 플러그인/MCP가 작업에 맞으면 적극 활용한다.
- MCP 리소스나 앱 커넥터가 로컬 추정보다 정확한 경우 MCP를 우선한다.
- 프론트엔드 확인은 가능하면 Browser/Playwright로 실제 화면을 검증한다.
- 보안 검토, PR 검토, CI 수정, 문서/스프레드시트/프레젠테이션 작업은 해당 전용 스킬을 먼저 따른다.
- 최신 정보, 외부 서비스 상태, 법/정책/가격/릴리스처럼 바뀔 수 있는 정보는 웹이나 공식 출처로 확인한다.

## 서비스 개발 원칙

- 기존 코드 구조, 네이밍, 프레임워크, 테스트 스타일을 먼저 읽고 따른다.
- 요구사항이 비어 있으면 보수적으로 결정한다. 범위는 작게 유지한다.
- 사용자 변경사항을 되돌리지 않는다. 작업 중 발견한 기존 변경은 존중한다.
- 불필요한 리팩터링, 포맷 변경, 메타데이터 변경은 피한다.
- 새 추상화는 실제 중복 제거, 복잡도 감소, 기존 패턴과의 일치가 있을 때만 만든다.
- 외부 입력, 인증, 권한, 개인정보, 결제, 파일 업로드, 네트워크 경계는 보안 관점으로 검토한다.
- 장애 가능성이 있는 경로에는 명확한 에러 처리와 로깅을 둔다.
- 설정값, 비밀값, 환경변수는 코드에 하드코딩하지 않는다.
- API 계약, 데이터 스키마, 마이그레이션은 호환성과 롤백 가능성을 고려한다.

## 구현 절차

- 먼저 저장소 구조와 관련 파일을 읽는다.
- 검색은 `rg`와 `rg --files`를 우선 사용한다.
- 변경 전, 무엇을 바꿀지 짧게 사용자에게 알린다.
- 파일 수정을 수동으로 할 때는 `apply_patch`를 사용한다.
- 테스트나 빌드가 필요한 변경이면 가능한 범위에서 직접 실행한다.
- 실패하면 원인을 확인하고, 추측으로 고치지 않는다.
- 완료 전 `git diff` 또는 관련 파일을 확인해 변경 범위가 의도와 맞는지 점검한다.

## 테스트와 검증

- 기능 변경에는 해당 단위 테스트나 통합 테스트를 추가/수정한다.
- 버그 수정은 실패 재현 테스트를 우선 고려한다.
- 프론트엔드 변경은 렌더링, 반응형 레이아웃, 클릭/입력 흐름을 실제 브라우저에서 확인한다.
- 성능, 접근성, 보안에 영향이 있는 변경은 해당 위험을 별도 확인한다.
- 테스트를 실행하지 못한 경우, 이유와 미검증 범위를 최종 응답에 명확히 적는다.

## Git 규칙

- 사용자가 요청하지 않으면 커밋, 푸시, PR 생성은 하지 않는다.
- `git reset --hard`, `git checkout --`, 대량 삭제 같은 파괴적 명령은 명시 요청 없이는 실행하지 않는다.
- 커밋이 필요하면 변경 범위를 확인하고 의미 있는 한국어 커밋 메시지를 쓴다.
- 커밋 메시지는 항상 접두사를 붙인다.
- 커밋 메시지 형식은 `[접두사]한국어 요약`을 따른다.
- 예시: `[Feat]회원가입 기능 추가`, `[Fix]로그인 실패 처리 수정`, `[Docs]개발 규칙 문서화`.
- 접두사는 다음 기준으로 선택한다.
  - `[Feat]`: 새 기능 추가
  - `[Fix]`: 버그 수정
  - `[Refactor]`: 동작 변경 없는 구조 개선
  - `[Docs]`: 문서 변경
  - `[Test]`: 테스트 추가/수정
  - `[Chore]`: 빌드, 설정, 의존성, 기타 유지보수
  - `[Perf]`: 성능 개선
  - `[Security]`: 보안 개선
- 커밋은 가능한 작고 원자적으로 만든다. 한 커밋은 한 의도를 담는다.
- 커밋 전에는 `git diff --staged`로 포함 범위를 확인한다.
- 작업 중 unrelated 변경이 있어도 건드리지 않는다.

## 브랜치 전략

- 기본 전략은 저명한 오픈소스 프로젝트와 실무 팀에서 널리 쓰는 GitHub Flow를 따른다.
- `main` 브랜치는 항상 배포 가능한 안정 상태로 유지한다.
- 직접 `main`에 작업하지 않는다. 변경은 목적별 브랜치에서 진행한다.
- 기능, 수정, 문서 작업은 짧은 수명 브랜치로 만든다.
- 브랜치 이름은 `종류/짧은-설명` 형식을 따른다.
- 브랜치 이름은 영어 소문자와 하이픈을 우선 사용한다.
- 예시: `feat/signup`, `fix/login-error`, `docs/agent-rules`, `chore/setup-ci`.
- 큰 기능은 작은 PR 여러 개로 나눈다.
- PR은 리뷰와 테스트 통과 후 `main`에 병합한다.
- 병합 전 최신 `main`을 반영해 충돌을 해결한다.
- 배포 릴리스나 긴급 수정이 필요할 때만 `release/*`, `hotfix/*` 브랜치를 사용한다.
- 장기 유지되는 `develop` 브랜치는 기본으로 사용하지 않는다. 필요할 때만 사용자와 합의 후 도입한다.

## GitHub 브랜치 보호 설정

- `main` 브랜치는 GitHub Rulesets로 보호한다.
- Ruleset 이름은 `Protect main`을 사용한다.
- Enforcement status는 `Active`로 설정한다.
- Target branches는 `Include default branch` 또는 `main`으로 설정한다.
- `Require a pull request before merging`을 켠다.
- Required approvals는 `1`로 설정한다.
- `Dismiss stale pull request approvals when new commits are pushed`를 켠다.
- `Require conversation resolution before merging`을 켠다.
- `Require status checks to pass`를 켠다.
- 필수 status check는 `Validate`를 사용한다.
- `Validate` check가 아직 목록에 없으면 첫 PR에서 CI를 한 번 실행한 뒤 Ruleset에 추가한다.
- `Block force pushes`를 켠다.
- `Restrict deletions`를 켠다.
- `Require linear history`를 켠다.
- `Automatically request Copilot code review`는 끈다.
- Codex PR 리뷰는 GitHub Actions가 아니라 Codex repository 등록 방식으로 사용한다.
- `Do not allow bypassing the above settings`는 초기 개인 개발 단계에서는 선택 사항이다. 팀 개발이나 배포 안정화 후 켠다.

## Review guidelines

- 리뷰는 심각한 버그, 보안 위험, 데이터 손상 가능성, 테스트 누락을 우선한다.
- `worker`는 서버 없는 로컬 macOS GUI 앱이며, 향후 Windows 확장을 고려한다.
- 개인 데이터, 파일 내용, 비밀값이 외부로 전송되는 변경은 P1 이상으로 본다.
- 파일 작업 기능은 원본 손상, 덮어쓰기, 인코딩 손상 가능성을 반드시 검토한다.
- Tauri, React, TypeScript, Vite, Rust, pnpm 스택과 맞지 않는 변경을 확인한다.
- UI와 작업 로직이 과도하게 결합되면 확장성 위험으로 지적한다.
- QR코드 기능은 URL 입력, PNG 저장, SVG 저장, 이미지 클립보드 복사를 기준으로 검토한다.
- 테스트가 없는 기능 변경은 위험도와 함께 지적한다.
- 리뷰 코멘트는 한국어로 작성한다.
- 불필요한 스타일 지적보다 실제 동작, 유지보수, 보안 위험을 우선한다.

## 커뮤니케이션

- 진행 중에는 짧게 상태를 공유한다.
- 막히면 무엇이 막혔는지, 확인한 것, 가능한 다음 선택지를 말한다.
- 최종 응답에는 변경 파일, 핵심 변경, 실행한 검증, 남은 위험만 간결히 적는다.
- 사용자가 명령 출력이나 결과를 요청하면 핵심 결과를 한국어로 전달한다.

## 우선순위

1. 사용자 명시 지시
2. 이 `AGENTS.md`
3. 저장소 기존 코드와 문서
4. 일반 엔지니어링 관행

충돌이 있으면 더 구체적이고 더 최근의 지시를 따른다.
